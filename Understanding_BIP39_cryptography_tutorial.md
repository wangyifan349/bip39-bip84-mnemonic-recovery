## 1. Scope

BIP39 defines a mnemonic code for deterministic wallets. It specifies how to convert cryptographic entropy into a human-readable sequence of words, and how to convert that mnemonic sentence plus an optional passphrase into a 512-bit seed. BIP39 does not define private-key derivation, public keys, addresses, derivation paths, signing algorithms, transaction formats, account discovery, or blockchain-specific address encodings.

The exact BIP39 boundary is:

```text
Entropy -> Mnemonic Sentence -> Seed
```

The usual wallet architecture continues after BIP39:

```text
Seed -> BIP32 Master Key -> Child Keys -> Public Keys -> Addresses
```

This document focuses on the cryptographic structure of BIP39: entropy, uncertainty, checksum, valid mnemonic probability, enumeration cost, and seed derivation. It intentionally avoids operational advice and non-cryptographic discussion.

## 3. Core Definitions

| Term | Definition |
|---|---|
| Entropy | Uniformly random binary input generated by a cryptographically secure random number generator. |
| ENT | Entropy length in bits. BIP39 allows 128, 160, 192, 224, or 256 bits. |
| SHA-256 | Cryptographic hash function used by BIP39 only to derive checksum bits from entropy. |
| Checksum | The first `ENT / 32` bits of `SHA256(entropy)`, appended to the entropy before word encoding. |
| CS | Checksum length in bits. `CS = ENT / 32`. |
| Mnemonic sentence | Ordered sequence of BIP39 words encoding `entropy || checksum`. |
| Wordlist | Fixed list of 2048 words. Since `2048 = 2^11`, each word encodes exactly 11 bits. |
| Seed | 512-bit value derived from mnemonic and optional passphrase using PBKDF2-HMAC-SHA512. |
| Passphrase | Optional string used as part of the PBKDF2 salt. It is not a mnemonic word. |
| BIP32 | HD wallet standard that commonly consumes the BIP39 seed and derives a tree of keys. |
| Derivation path | Path notation such as `m/84'/0'/0'/0/0`, used after BIP39 by BIP32-style wallets. |

## 4. Entropy and Uncertainty

In cryptography, entropy measures uncertainty. If a value is uniformly random over `n` bits, the number of possible values is:

```text
2^n
```

A 128-bit entropy value has:

```text
2^128 = 340,282,366,920,938,463,463,374,607,431,768,211,456
```

possible values.

A 256-bit entropy value has:

```text
2^256 = 115,792,089,237,316,195,423,570,985,008,687,907,853,269,984,665,640,564,039,457,584,007,913,129,639,936
```

possible values.

BIP39 mnemonic security is based on the entropy length, not the number of words alone. The checksum adds structure and error detection, but it does not add independent entropy.

| Word Count | ENT | CS | Total Encoded Bits | Security Strength |
|---:|---:|---:|---:|---:|
| 12 | 128 | 4 | 132 | 128-bit |
| 15 | 160 | 5 | 165 | 160-bit |
| 18 | 192 | 6 | 198 | 192-bit |
| 21 | 224 | 7 | 231 | 224-bit |
| 24 | 256 | 8 | 264 | 256-bit |

The correct security strength is `ENT`, not `ENT + CS`.

## 5. Mnemonic Space vs Valid Mnemonic Space

A BIP39 wordlist has 2048 words. Therefore, any sequence of `MS` words has a raw sequence space of:

```text
2048^MS = (2^11)^MS = 2^(11MS)
```

However, not every word sequence is a valid BIP39 mnemonic. The checksum imposes a validity constraint. For each valid entropy value, BIP39 produces exactly one valid mnemonic sentence.

Therefore, the number of valid BIP39 mnemonics is:

```text
2^ENT
```

The number of raw word sequences is:

```text
2^(ENT + CS)
```

The probability that a uniformly random word sequence of the correct length passes checksum validation is:

```text
2^ENT / 2^(ENT + CS) = 2^(-CS)
```

| Words | Raw Word Sequences | Valid Mnemonics | Checksum Bits | Probability Random Sequence Is Valid |
|---:|---:|---:|---:|---:|
| 12 | `2^132` | `2^128` | 4 | `1/16 = 6.25%` |
| 15 | `2^165` | `2^160` | 5 | `1/32 = 3.125%` |
| 18 | `2^198` | `2^192` | 6 | `1/64 = 1.5625%` |
| 21 | `2^231` | `2^224` | 7 | `1/128 = 0.78125%` |
| 24 | `2^264` | `2^256` | 8 | `1/256 = 0.390625%` |

This table is important: a 24-word mnemonic encodes 264 total bits, but only 256 bits are entropy. The remaining 8 bits are checksum.

## 6. Checksum

BIP39 checksum is computed as follows:

```text
checksum = first ENT / 32 bits of SHA256(entropy)
```

Then:

```text
encoded_bits = entropy || checksum
```

The checksum is a deterministic function of the entropy. It is not random independent data and does not increase brute-force security. Its purpose is structural validation.

### Checksum Length

| ENT | CS |
|---:|---:|
| 128 | 4 |
| 160 | 5 |
| 192 | 6 |
| 224 | 7 |
| 256 | 8 |

### Checksum Passing Probability

For a uniformly random candidate mnemonic of the correct word count, the chance of passing checksum validation is:

```text
2^(-CS)
```

So:

```text
12 words: 1 / 16
24 words: 1 / 256
```

This is a validation probability, not a wallet-compromise probability. Passing checksum means only that the mnemonic has valid BIP39 structure. It does not mean the mnemonic corresponds to a wallet with funds.

### Checksum and Error Detection

If an accidental modification produces a uniformly random checksum relation, the probability that the modified mnemonic still passes checksum is approximately:

```text
2^(-CS)
```

For 12 words, approximately `1/16` modified candidates may pass checksum. For 24 words, approximately `1/256` modified candidates may pass checksum. The checksum is short; it is not a cryptographic authentication code.

## 7. Wordlist and 11-bit Encoding

Because:

```text
2048 = 2^11
```

each mnemonic word corresponds to exactly one 11-bit value.

```text
00000000000 -> index 0
00000000001 -> index 1
00000000010 -> index 2
...
11111111111 -> index 2047
```

The wordlist is an ordered table, not a semantic dictionary. The word meaning is irrelevant to the cryptographic process. Only the index matters.

Example:

```text
index 0    -> abandon
index 1    -> ability
index 2    -> able
...
index 2047 -> zoo
```

A valid implementation must use the exact wordlist and exact ordering for the selected language. Reordering the wordlist changes the mapping and produces incompatible mnemonics.

## 8. Mnemonic Generation Algorithm

The generation algorithm is deterministic after entropy is chosen.

```text
Input:
  entropy bytes
  BIP39 wordlist

Steps:
  1. Verify entropy length ENT ∈ {128,160,192,224,256}.
  2. Compute hash = SHA256(entropy).
  3. Compute CS = ENT / 32.
  4. Take the first CS bits of hash.
  5. Append checksum bits to entropy bits.
  6. Split the result into 11-bit groups.
  7. Convert each group to an integer index.
  8. Select wordlist[index] for each group.
  9. Join selected words into the mnemonic sentence.
```

Compact notation:

```text
entropy -> SHA256 -> checksum
entropy || checksum -> 11-bit groups -> word indices -> mnemonic
```

## 9. Mnemonic Validation Algorithm

Validation reverses the encoding and checks the checksum.

```text
Input:
  mnemonic sentence
  BIP39 wordlist

Steps:
  1. Split the mnemonic into words.
  2. Confirm word count ∈ {12,15,18,21,24}.
  3. Convert each word to its wordlist index.
  4. Convert each index to an 11-bit binary group.
  5. Concatenate all groups.
  6. Split into entropy bits and checksum bits.
  7. Compute SHA256(entropy).
  8. Compare extracted checksum with expected checksum.
```

For `MS` words:

```text
total_bits = 11 * MS
CS = total_bits / 33
ENT = total_bits - CS
```

Example for 24 words:

```text
total_bits = 11 * 24 = 264
CS = 264 / 33 = 8
ENT = 264 - 8 = 256
```

## 10. Seed Derivation

After the mnemonic sentence is available, BIP39 derives a 512-bit seed using PBKDF2-HMAC-SHA512.

Parameters:

| Parameter | Value |
|---|---|
| Function | PBKDF2 |
| PRF | HMAC-SHA512 |
| Password | `NFKD(mnemonic)` |
| Salt | `NFKD("mnemonic" + passphrase)` |
| Iterations | 2048 |
| Output length | 64 bytes |
| Output strength | Determined by input entropy and passphrase entropy |

Formula:

```text
seed = PBKDF2-HMAC-SHA512(
    password = NFKD(mnemonic),
    salt = NFKD("mnemonic" + passphrase),
    iterations = 2048,
    dkLen = 64
)
```

The 512-bit seed is not equivalent to 512 bits of security. The security is bounded by the entropy of the mnemonic and passphrase. For a standard 12-word mnemonic generated from 128-bit entropy, the mnemonic contributes 128 bits of security. For a standard 24-word mnemonic generated from 256-bit entropy, the mnemonic contributes 256 bits of security.

## 11. Passphrase and Seed Separation

The passphrase is optional and changes the derived seed.

```text
same mnemonic + passphrase A -> seed A
same mnemonic + passphrase B -> seed B
```

The passphrase is not encoded into the mnemonic. It is used only during seed derivation.

```text
salt = "mnemonic" + passphrase
```

If the passphrase is empty:

```text
salt = "mnemonic"
```

If the passphrase is `alpha`:

```text
salt = "mnemonicalpha"
```

The passphrase adds security only to the extent that it has entropy. A short or predictable passphrase has low entropy. A uniformly random passphrase over a large space can add substantial search complexity.

## 12. BIP39 and HD Wallets

BIP39 usually feeds BIP32.

```text
BIP39 seed -> BIP32 master node -> child keys
```

BIP32 creates a hierarchical deterministic wallet tree.

```text
m
├── m/0
├── m/1
└── m/2
    ├── m/2/0
    └── m/2/1
```

The BIP39 seed is used as input to BIP32 master key generation. BIP32 then derives extended private keys and extended public keys. BIP39 itself does not define the child-key tree or any address format.

## 13. Derivation Paths

A derivation path selects a specific branch inside a BIP32 HD wallet tree. BIP39 does not define paths, but BIP39 wallets commonly use standard path conventions.

General BIP44-style form:

```text
m / purpose' / coin_type' / account' / change / address_index
```

| Component | Meaning |
|---|---|
| `m` | Master node. |
| `purpose'` | Standard namespace, hardened. |
| `coin_type'` | Coin identifier, hardened. Bitcoin mainnet is commonly `0'`; testnet is commonly `1'`. |
| `account'` | Account index, hardened. |
| `change` | `0` for external receiving addresses, `1` for internal change addresses. |
| `address_index` | Sequential address index. |

Common examples:

| Standard | Path | Script / Address Type |
|---|---|---|
| BIP44 | `m/44'/0'/0'/0/0` | Bitcoin Legacy P2PKH |
| BIP49 | `m/49'/0'/0'/0/0` | Nested SegWit P2SH-P2WPKH |
| BIP84 | `m/84'/0'/0'/0/0` | Native SegWit P2WPKH |
| BIP86 | `m/86'/0'/0'/0/0` | Taproot P2TR |
| Ethereum common path | `m/44'/60'/0'/0/0` | Ethereum account key |

A mnemonic and passphrase determine the seed. The derivation path determines which keys and addresses are derived from that seed.

## 14. Cryptographic Strength

The cryptographic strength of a BIP39 mnemonic generated correctly is determined by the entropy length.

| Words | ENT | Approximate Security Class |
|---:|---:|---|
| 12 | 128 bits | Strong |
| 15 | 160 bits | Very strong |
| 18 | 192 bits | Very strong |
| 21 | 224 bits | Extremely strong |
| 24 | 256 bits | Extremely strong |

The checksum does not increase cryptographic strength. It filters valid encodings.

Correct statement:

```text
12-word BIP39 mnemonic: 128-bit entropy + 4-bit checksum = 132 encoded bits, 128-bit security.
24-word BIP39 mnemonic: 256-bit entropy + 8-bit checksum = 264 encoded bits, 256-bit security.
```

The 512-bit seed output does not imply 512-bit mnemonic security. A KDF cannot create more entropy than its inputs contain.

## 15. Exhaustive Search Time

For a search rate `R` candidates per second, exhaustive search time is:

```text
T_seconds = 2^ENT / R
T_years = 2^ENT / (R * 31,556,952)
```

where `31,556,952` is the average number of seconds in a Gregorian year.

The table below shows full-space enumeration time, not average success time. Average success time for a uniformly random target is approximately half the full-space time.

| ENT | `10^6/s` | `10^9/s` | `10^12/s` | `10^15/s` | `10^18/s` |
|---:|---:|---:|---:|---:|---:|
| 128 | `1.078e25` years | `1.078e22` years | `1.078e19` years | `1.078e16` years | `1.078e13` years |
| 160 | `4.631e34` years | `4.631e31` years | `4.631e28` years | `4.631e25` years | `4.631e22` years |
| 192 | `1.989e44` years | `1.989e41` years | `1.989e38` years | `1.989e35` years | `1.989e32` years |
| 224 | `8.543e53` years | `8.543e50` years | `8.543e47` years | `8.543e44` years | `8.543e41` years |
| 256 | `3.669e63` years | `3.669e60` years | `3.669e57` years | `3.669e54` years | `3.669e51` years |

These estimates assume idealized direct candidate testing at the stated rate. In real BIP39 attacks, each candidate typically requires checksum handling, PBKDF2-HMAC-SHA512 seed derivation, and downstream wallet/address checks. The exact throughput depends on hardware, implementation, passphrase assumptions, target chain, and known address information.

## 16. Exact Probability Statements

### 16.1 Probability of Guessing a Specific Mnemonic

For a properly generated mnemonic with `ENT` bits of entropy, the probability of guessing it in one uniformly random entropy attempt is:

```text
1 / 2^ENT
```

Examples:

```text
12 words: 1 / 2^128
24 words: 1 / 2^256
```

### 16.2 Probability a Random Word Sequence Is Structurally Valid

For a random word sequence of the correct length using the BIP39 wordlist:

```text
P(valid checksum) = 2^(-CS)
```

Examples:

```text
12 words: 1 / 16
24 words: 1 / 256
```

### 16.3 Probability a Valid Random Mnemonic Equals a Target Mnemonic

If sampling uniformly from valid mnemonics of the same entropy class:

```text
P(match target) = 1 / 2^ENT
```

### 16.4 Probability a Raw Random Word Sequence Equals a Target Mnemonic

If sampling uniformly from all raw word sequences of length `MS`:

```text
P(match target) = 1 / 2048^MS = 1 / 2^(11MS)
```

For 12 words:

```text
1 / 2048^12 = 1 / 2^132
```

This differs from `1 / 2^128` because most raw 12-word sequences are not checksum-valid. If the attack process only tests valid BIP39 mnemonics, the relevant target space is `2^128`.

### 16.5 Probability of Passing Checksum and Matching a Target

For raw random 12-word sequences:

```text
P(valid) = 1 / 16
P(match target | valid) = 1 / 2^128
P(match target) = 1 / 2^132
```

For valid generated 12-word mnemonics:

```text
P(match target) = 1 / 2^128
```

The correct probability depends on the sampling model.

## 17. Reference Pseudocode

### 17.1 Entropy to Mnemonic

```text
function entropy_to_mnemonic(entropy_bytes, wordlist):
    ENT = bit_length(entropy_bytes)

    if ENT not in [128, 160, 192, 224, 256]:
        error("invalid entropy length")

    hash_bytes = SHA256(entropy_bytes)
    CS = ENT / 32

    entropy_bits = bytes_to_bits_msb_first(entropy_bytes)
    hash_bits = bytes_to_bits_msb_first(hash_bytes)

    checksum_bits = first_bits(hash_bits, CS)
    combined_bits = entropy_bits || checksum_bits

    groups = split_into_groups(combined_bits, 11)

    words = []
    for group in groups:
        index = bits_to_integer(group)
        words.append(wordlist[index])

    return join(words, " ")
```

### 17.2 Mnemonic Validation

```text
function validate_mnemonic(mnemonic, wordlist):
    words = split_words(mnemonic)

    if len(words) not in [12, 15, 18, 21, 24]:
        return false

    bits = ""

    for word in words:
        if word not in wordlist:
            return false

        index = wordlist_index(word)
        bits += integer_to_11_bit_binary(index)

    total_bits = len(bits)

    CS = total_bits / 33
    ENT = total_bits - CS

    entropy_bits = bits[0:ENT]
    checksum_bits = bits[ENT:ENT+CS]

    entropy_bytes = bits_to_bytes(entropy_bits)

    hash_bits = bytes_to_bits_msb_first(SHA256(entropy_bytes))
    expected_checksum = first_bits(hash_bits, CS)

    return checksum_bits == expected_checksum
```

### 17.3 Mnemonic to Seed

```text
function mnemonic_to_seed(mnemonic, passphrase):
    normalized_mnemonic = NFKD(mnemonic)
    normalized_passphrase = NFKD(passphrase)

    password = UTF8(normalized_mnemonic)
    salt = UTF8("mnemonic" + normalized_passphrase)

    seed = PBKDF2_HMAC_SHA512(
        password = password,
        salt = salt,
        iterations = 2048,
        output_length = 64
    )

    return seed
```

## 18. Implementation Requirements

A rigorous BIP39 implementation should satisfy the following checklist:

```text
[ ] Use CSPRNG-generated entropy.
[ ] Accept only ENT ∈ {128,160,192,224,256}.
[ ] Compute checksum as first ENT/32 bits of SHA256(entropy).
[ ] Append checksum bits to entropy bits.
[ ] Split combined bits into exact 11-bit groups.
[ ] Use an exact 2048-word BIP39 wordlist.
[ ] Preserve wordlist order.
[ ] Validate mnemonic length.
[ ] Validate every word against the selected wordlist.
[ ] Reconstruct entropy and checksum from word indices.
[ ] Verify checksum.
[ ] Apply UTF-8 NFKD normalization before PBKDF2.
[ ] Use PBKDF2-HMAC-SHA512 with 2048 iterations.
[ ] Output exactly 64 bytes of seed.
[ ] Verify behavior against official BIP39 test vectors.
```

## 19. Glossary

**BIP39**: A standard defining mnemonic code generation and seed derivation for deterministic wallets.

**Entropy**: Random binary input. In BIP39, entropy is the true source of mnemonic security.

**Uncertainty**: The attacker's lack of knowledge about the entropy value. For `n` uniformly random bits, uncertainty is `2^n` possibilities.

**Checksum**: Short deterministic bit sequence derived from `SHA256(entropy)`. It validates structure but does not add independent entropy.

**Mnemonic Sentence**: Ordered sequence of words encoding `entropy || checksum`.

**Wordlist**: Ordered list of 2048 words. Each word corresponds to one 11-bit index.

**Seed**: 512-bit PBKDF2-HMAC-SHA512 output derived from mnemonic and optional passphrase.

**Passphrase**: Optional string included in the PBKDF2 salt as `"mnemonic" + passphrase`.

**CSPRNG**: Cryptographically Secure Random Number Generator.

**SHA-256**: Cryptographic hash function used for checksum construction.

**PBKDF2**: Password-Based Key Derivation Function 2.

**HMAC-SHA512**: HMAC construction using SHA-512, used inside PBKDF2.

**BIP32**: HD wallet standard that derives master and child keys from a seed.

**Derivation Path**: A path such as `m/84'/0'/0'/0/0` specifying a position in a BIP32 key tree.

**Hardened Derivation**: BIP32 derivation marked by `'`, such as `84'`, which uses hardened child-key derivation.

**Security Strength**: Effective brute-force resistance measured in bits, determined primarily by entropy length.
